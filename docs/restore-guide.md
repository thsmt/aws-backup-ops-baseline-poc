# 復旧確認

## 概要

バックアップ運用では、バックアップジョブの成功だけでなく、復旧できることを定期的に確認する必要があります。

このドキュメントでは、AWS Backup の復旧確認で見るべき観点を整理します。

<br>

## 復旧確認の流れ

1. Backup Vault から復旧ポイントを選択します。
2. 復旧対象のリソース種別を確認します。
3. 復旧先の VPC、Subnet、Security Group、IAM Role などを確認します。
4. 復旧ジョブを開始します。
5. 復旧後のリソースに接続し、起動状態やデータ整合性を確認します。
6. 復旧確認後、検証用に作成したリソースを削除します。

<br>

## 確認観点

| 観点 | 内容 |
| --- | --- |
| 復旧ポイント | 想定した日時の復旧ポイントが存在するか |
| 復旧先 | 既存環境に影響しない検証用の復旧先か |
| ネットワーク | 復旧後に必要な接続経路があるか |
| セキュリティ | Security Group や IAM Role が過剰になっていないか |
| アプリケーション | サービス起動、ログイン、データ参照ができるか |
| 後片付け | 検証用リソースを削除したか |

<br>

## EC2 / EBS の復旧確認

EC2 や EBS の復旧では、復旧後のインスタンスまたはボリュームを確認します。

確認例:

- インスタンスが起動すること
- OSログに異常がないこと
- 必要なボリュームがアタッチされていること
- アプリケーションプロセスが起動していること

<br>

## RDS の復旧確認

RDS の復旧では、既存DBへ上書きせず、新しいDBインスタンスとして復旧する運用が基本です。

確認例:

- DBインスタンスが利用可能状態になること
- 接続できること
- テーブルや主要データが存在すること
- アプリケーション接続設定との差分を把握できること

<br>

## 復旧テストの記録

復旧確認を実施したら、以下を記録します。

| 項目 | 内容 |
| --- | --- |
| 実施日 | 復旧確認を行った日 |
| 対象リソース | 復旧対象のリソース |
| 復旧ポイント | 使用した復旧ポイント |
| 結果 | 成功 / 失敗 |
| 所要時間 | 復旧開始から確認完了までの時間 |
| 備考 | 問題点、改善点 |
